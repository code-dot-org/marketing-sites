# Feature Specification: Sitewide Typography System

**Feature Branch**: `009-typography-system`
**Created**: 2026-06-23
**Status**: Draft
**Input**: User description: "We need to update the fonts across the site. This will entail a standardized, MUI-based, sizing system… 4 weights × 8 sizes per font (Space Grotesk headings, Geist body). Noto Sans i18n fallback chain. H1 = Display 2xl Semibold; body default = Text Md Medium (Geist, 16px, weight 500). Heading + Paragraph keep overrides but render canonical defaults. Deprecated components (Hero Banner) reflect the canonical heading defaults. Responsive scaling per viewport, floored at 16px body. All values in REM. code.org only — CSforAll not affected." Subsequent amendments (2026-06-23): (a) body line-height pinned to the Figma `text-md` token (1.5rem / 24px), overriding the original 1.25rem / 20px statement; (b) **Heading Level is decoupled from Visual Appearance** — the existing Heading-Level dropdown stays as the `<h1>`–`<h6>` semantic-tag selector, a new "Visual Appearance" dropdown is added with the 8 Display cells as size-override options (each carrying its canonical default style), and Paragraph's existing Visual Appearance enum is widened to all 8 Text cells. Precedence resolves Heading Level default → Visual Appearance override → individual field overrides.

## Overview

This feature replaces the current ad-hoc typography setup with a single, MUI-based typography token system for the **code.org** tenant. It rebuilds the existing MUI typography variants and the SCSS font-size primitives around two tracks — a **Display** track in **Space Grotesk** (for headings) and a **Text** track in **Geist** (for body, lists, captions, link labels) — each exposing **4 weights × 8 sizes** (`xs, sm, md, lg, xl, 2xl, 3xl, 4xl`). It re-introduces the **Noto Sans fallback chain** in front of the generic `sans-serif` keyword so non-Latin glyphs render in a consistent, locale-aware family rather than the browser default.

Every HTML heading level (`H1`–`H6`) and the body default are bound to a **canonical role token** in the MUI theme (`theme.typography.h1` … `theme.typography.h6`, `theme.typography.body1`). Each role token resolves to one cell in the (track × size × weight) grid — for example, **H1 = Display 2xl Semibold**; **body default = Text Md Medium** (Geist, 1rem / 16px, weight 500, line-height 1.5rem / 24px per the Figma `text-md` token). Changing what H1 looks like across the site becomes a one-line edit to the role token.

The **Heading** Contentful component grows a second orthogonal dropdown so authors can decouple semantic level from visual size:

- **Heading Level** (the existing `visualAppearance` field, Studio displayName "Heading Level") continues to select `<h1>` through `<h6>` and provides the **canonical default size + weight + line-height + letter-spacing + per-breakpoint steps** for that level.
- **Visual Appearance** (a new field, internal name `appearance`, Studio displayName "Visual Appearance") is a sentinel `default` plus the 8 Display cells (`display-xs` through `display-4xl`). Picking a Display cell **overrides the size, weight, line-height, letter-spacing, and per-breakpoint step table** from the Heading Level's canonical role with the chosen cell's canonical role. The Display track and the rendered `<h*>` semantic tag are unchanged — Visual Appearance ONLY affects the size/style cell.
- **Individual override fields** (`fontSize`, `fontWeight`, `lineHeight`, `fontKerning`, `textTransform`, `colorOverride`, `zIndex`) take **top precedence** and override whichever cell the Visual Appearance / Heading Level chain settled on.

The **Paragraph** Contentful component keeps its existing **Visual Appearance** dropdown but **widens its enum to include all 8 Text cells** (`text-xs` through `text-4xl`) alongside the legacy `body-one`/`body-two`/`body-three`/`body-four` values. The legacy values retain their existing role-token bindings; the new values map to fresh role tokens (Text cell + canonical weight) so authors get the full Text scale. Default stays `body-two` (Text md Medium = body default).

Deprecated components that hardcode their own type today — **Hero Banner**, `YourSchool`, `AdoptionMap`, `AfeEligibility`, and any other code.org-only component bypassing Heading/Paragraph — MUST consume the same role tokens so that a default change in one place propagates everywhere.

**Responsive scaling**: each role token expresses a per-breakpoint size mapping using the 8-step scale (`xs`–`4xl`) rather than a continuous clamp. At narrower viewports a heading steps down through the scale; the floor is the body base (`1rem` / 16px) so the smallest heading at the smallest viewport never falls below body text. Hierarchy (H1 visually larger than H2, etc.) is preserved at every breakpoint.

**Tenant scope**: code.org only. The CSforAll typography theme (`apps/marketing/src/themes/csforall/index.ts`) — including its Roboto Mono body and 800-weight headings — is **not modified**. CSforAll continues to render unchanged.

**Deprecated-component policy** (resolved at spec time): The cheap path — tie deprecated components to the new role tokens via shared theme variants rather than re-engineering them — is in scope. No additional "heading default" token type beyond the standard MUI `theme.typography.h1…h6` variants is required, because that variant layer is itself the per-heading-default token surface: it accepts a full `{ fontFamily, fontSize, fontWeight, lineHeight, letterSpacing, [breakpoints] }` payload and is already the MUI-native single-source-of-truth for heading-level defaults. Adding a parallel custom-token layer (e.g. `--codeai-heading-h1: …`) would duplicate that contract without adding capability.

## User Scenarios & Testing _(mandatory)_

### User Story 1 — Heading and Paragraph Defaults Resolve From a Single Source (Priority: P1)

A frontend developer or content author needs every `H1` on the code.org site to look the same: Space Grotesk, Display 2xl size, Semibold weight, with consistent line-height and letter-spacing. They place a Heading entry in Contentful with `level = h1` and leave every other field at its default. The rendered page shows the canonical H1 treatment. They open a different page that also has a default H1 and see the identical treatment. They then change the H1 role token in the code.org typography theme from "Display 2xl Semibold" to "Display 3xl Semibold"; every default H1 across the site updates without per-entry edits.

**Why this priority**: This is the entire payload. Without canonical role tokens that components consume by default, the typography stays inconsistent and per-entry overrides multiply across Contentful. Every other story builds on this one.

**Independent Test**: In Storybook, the Heading component's "default" story for each level (`h1`–`h6`) renders the canonical role token. In a marketing preview page, two Heading entries with `level = h1` and no other props render identically. Change `theme.typography.h1` in `apps/marketing/src/themes/code.org/index.ts` and confirm both rendered H1s pick up the new size/weight without re-publishing the entries.

**Acceptance Scenarios**:

1. **Given** a Heading entry with `level = h1` and no overrides, **When** the page renders on code.org, **Then** the heading renders as Space Grotesk, Display 2xl size, Semibold weight, with the line-height and letter-spacing defined in the H1 role token.
2. **Given** a Paragraph entry with no `visualAppearance` override, **When** the page renders, **Then** the body text renders as Geist, Text Md, weight 500, line-height 1.5rem (24px) per the Figma `text-md` token.
3. **Given** two Heading entries on different pages with `level = h2` and no overrides, **When** both pages render, **Then** both H2s are visually identical in every typographic dimension (font, size, weight, line-height, letter-spacing) at every breakpoint.
4. **Given** the H1 role token is changed from Display 2xl Semibold to Display 3xl Semibold, **When** the site is rebuilt, **Then** every default H1 across every code.org page renders at the new token without per-entry edits.
5. **Given** a Heading entry with `level = h3` and an explicit `weight = bold` override, **When** the page renders, **Then** the size and font-family come from the H3 role token and only the weight is overridden.

### User Story 2 — Deprecated Components Reflect the Canonical Heading Defaults (Priority: P1)

A content author uses an in-use deprecated component such as **Hero Banner** that hardcodes its title typography today. After this feature ships, that component's title renders at the same canonical H1 (or equivalent) role token as the new Heading component. When the H1 default is later edited centrally, the Hero Banner title updates with the rest of the site — without any per-component code change.

**Why this priority**: A central token surface is only meaningful if the deprecated surface area routes through it. Otherwise, every default change requires hunting down hardcoded font sizes, undoing the "single source of truth" goal. Until those components are removed, they MUST consume the same role tokens.

**Independent Test**: Identify each code.org deprecated component that currently bypasses Heading/Paragraph (Hero Banner, `YourSchool`, `AdoptionMap`, `AfeEligibility`, and any others found in research). For each, confirm that its title/body typography routes through the new MUI theme variants or shared font-size CSS variables. Edit the H1 role token; confirm every such component picks up the change in Storybook.

**Acceptance Scenarios**:

1. **Given** a Hero Banner entry in Contentful, **When** the page renders, **Then** the banner title typography matches the canonical H1 role token (Space Grotesk, Display 2xl Semibold, line-height/letter-spacing from the token).
2. **Given** `YourSchool`, `AdoptionMap`, and `AfeEligibility` components, **When** they render their internal headings and body text, **Then** their typography references the new MUI variants or shared SCSS font-size variables — not inline hardcoded `font-size: 2rem` / `font-weight: 500` / `font-size: 0.75rem` style rules.
3. **Given** the H1 role token is changed, **When** the site is rebuilt, **Then** the Hero Banner title and any other deprecated-component "primary heading" surface updates without component-level edits.
4. **Given** any deprecated component is later removed, **When** the removal happens, **Then** no token cleanup is required because no token was introduced specifically for that component.

### User Story 3 — Authors Decouple Heading Level From Visual Appearance (Priority: P1)

A content author wants an `<h2>` (for the page's document outline / SEO / accessibility) that visually reads like the canonical H1 (Display 2xl Semibold). They open a Heading entry in Studio: they set **Heading Level = "Heading 2"** and **Visual Appearance = "Display 2xl"**. The rendered HTML is `<h2>` (preserving the outline) but the visual size, weight, line-height, and letter-spacing are the canonical H1 cell. They can also do the inverse — set Heading Level = "Heading 1" but Visual Appearance = "Display lg" — to get an `<h1>` that's visually smaller. Across the page, no other H2 entry is affected.

**Why this priority**: This is the orthogonal-field model the user asked for. Without it, authors must misuse the heading-level dropdown to achieve a visual size (sacrificing document outline / SEO / a11y). With it, the semantic tag and the visual treatment are independent levers, the canonical role token is the default, and individual override fields stay as the escape hatch. This is foundational for how Heading is used going forward.

**Independent Test**: In Storybook, render four Heading entries: (A) Heading Level = "Heading 1", Visual Appearance = "default" — confirms canonical H1 (Display 2xl Semibold) inside an `<h1>` tag. (B) Heading Level = "Heading 2", Visual Appearance = "Display 2xl" — confirms `<h2>` tag with the same Display 2xl Semibold size. (C) Heading Level = "Heading 1", Visual Appearance = "Display lg" — confirms `<h1>` tag with Display lg Semibold size. (D) Heading Level = "Heading 3", Visual Appearance = "default", `fontWeight = '700'` — confirms `<h3>` tag with canonical H3 size and weight 700 (override wins).

**Acceptance Scenarios**:

1. **Given** a Heading entry with Heading Level = "Heading 2" and Visual Appearance = "default" (no override), **When** the page renders, **Then** the rendered tag is `<h2>` and the visual treatment matches the canonical H2 role token (Display xl Semibold + H2's per-breakpoint steps).
2. **Given** a Heading entry with Heading Level = "Heading 2" and Visual Appearance = "Display 2xl", **When** the page renders, **Then** the rendered tag is still `<h2>` and the visual treatment matches the Display 2xl role token (Display 2xl Semibold + Display 2xl's per-breakpoint steps).
3. **Given** a Heading entry with Heading Level = "Heading 1" and Visual Appearance = "Display lg", **When** the page renders, **Then** the rendered tag is `<h1>` and the visual treatment matches Display lg Semibold — smaller than the canonical H1.
4. **Given** a Heading entry with Heading Level = "Heading 3", Visual Appearance = "Display xl", and individual `fontSize = 5` (rem), **When** the page renders, **Then** the rendered tag is `<h3>`, the family is Display, the line-height + letter-spacing come from the Display xl cell, and `font-size: 5rem` from the individual override wins.
5. **Given** a Paragraph entry with Visual Appearance = "text-2xl", **When** the page renders, **Then** the paragraph renders at Text 2xl Medium (1.5rem / 2rem line-height) — a new value not previously available via the `body-one`–`body-four` list.
6. **Given** existing Heading entries with `visualAppearance` set to `heading-xxl`/`xl`/`lg`/`md`/`sm`/`xs` and no `appearance` field value (the new field is absent), **When** the page renders, **Then** the entries continue to render exactly as they did pre-feature — semantic tag from the Heading Level enum (legacy 1:1 mapping), visual treatment from the canonical role token for that level. No author re-publish required.
7. **Given** existing Paragraph entries with `visualAppearance` set to `body-one`/`body-two`/`body-three`/`body-four`, **When** the page renders, **Then** the entries continue to render with their existing role-token bindings (Text lg Medium / Text md Medium / Text sm Regular / Text xs Regular). No author re-publish required.

### User Story 4 — Authors Can Override Per Instance Without Breaking Defaults (Priority: P2)

A content author needs a one-off H2 rendered at a custom font weight, while every other H2 on the site stays at the canonical H2. They edit the Heading entry's existing individual override fields (e.g. `fontWeight = '700'`) in Contentful; the page renders the override; every other default H2 is unaffected. Same flow for Paragraph: tweak `fontSize` / `fontWeight` on one Paragraph entry without affecting the body default elsewhere.

**Why this priority**: Defaults and the new Visual Appearance override are the rule, but real marketing pages need occasional exceptions that fall outside the role-token grid. Heading and Paragraph already expose per-instance individual override fields; this feature keeps those override surfaces and locks them as **top precedence** above both the Heading Level default AND the Visual Appearance override. Without them, the feature would force authors back into custom code for any out-of-grid tweak.

**Independent Test**: In Storybook, render four Heading entries: (A) Heading Level = "Heading 2", no other props — confirms canonical H2. (B) Heading Level = "Heading 2", individual `fontSize = 5` (rem) — confirms `<h2>` with `font-size: 5rem` and the rest of the canonical H2 carried through. (C) Heading Level = "Heading 2", Visual Appearance = "Display xl", individual `fontWeight = '700'` — confirms `<h2>` with Display xl size + line-height + letter-spacing and weight 700 (Bold). (D) Paragraph with `visualAppearance = "text-lg"` and individual `isStrong = true` — confirms Text lg + Semibold weight.

**Acceptance Scenarios**:

1. **Given** a Heading entry with Heading Level = "Heading 2", Visual Appearance = "default", and an individual `fontSize = 5` (rem) override, **When** the page renders, **Then** the `<h2>` renders at `font-size: 5rem` while line-height, letter-spacing, family, and weight come from H2's canonical role token.
2. **Given** a Heading entry with Heading Level = "Heading 2", Visual Appearance = "Display xl", and an individual `fontWeight = '700'` override, **When** the page renders, **Then** the `<h2>` renders with Display xl's size + line-height + letter-spacing (from the Visual Appearance override) and weight 700 (from the individual override).
3. **Given** a Paragraph entry with `visualAppearance = "text-lg"`, `isStrong = true`, and an individual `colorOverride = '#FF0000'`, **When** the page renders, **Then** the paragraph renders at Text lg size + line-height (from the Visual Appearance), Semibold weight (from `isStrong`), and red color (from the override).
4. **Given** an individual override is set at the entry level, **When** the canonical role token is later edited, **Then** the overridden entry keeps the override; only the unspecified parts (family, unset fields) inherit from whichever role token the Visual Appearance / Heading Level chain settled on.
5. **Given** the Heading.fontWeight Contentful enum is still `['500', '700']` (Studio surface), **When** an author needs Regular (400) or Semibold (600), **Then** they pick a Visual Appearance whose canonical default already carries that weight, OR the override stays unavailable in Studio for this feature (the 4-weight Studio ladder is a deferred follow-up per FR-024 commentary).

### User Story 5 — Noto Sans Fallback Chain Restores i18n Coverage (Priority: P2)

A page renders content that includes characters Space Grotesk and Geist do not cover — Arabic, Devanagari, Japanese, Korean, Simplified Chinese, Traditional Chinese, etc. The browser falls through the font stack to a locale-appropriate Noto Sans family **before** reaching the generic `sans-serif` keyword, so all characters render in a coherent typographic style rather than the OS default sans for non-Latin scripts.

**Why this priority**: The Noto Sans chain existed before recent typography work and was lost. Without it, mixed-script content (especially in CJK locales) renders in browser-default fonts that clash with the brand body and headings.

**Independent Test**: Render a page containing Latin + Arabic + CJK + Devanagari text using the Heading and Paragraph components. With Space Grotesk / Geist intentionally unavailable (DevTools "Network throttling" or font-family override), confirm the non-Latin glyphs render in the appropriate Noto Sans family (not in Times/serif default or system default), and that the chain ends in `sans-serif` as final fallback.

**Acceptance Scenarios**:

1. **Given** any Heading rendering on code.org, **When** the resolved font-stack is inspected, **Then** Space Grotesk is first, followed by the Noto Sans family list (CJK JP/SC/TC, KR, Arabic, Devanagari, Latin, etc.), with `sans-serif` last.
2. **Given** any Paragraph or Text-track element on code.org, **When** the resolved font-stack is inspected, **Then** Geist is first, followed by the same Noto Sans family list, with `sans-serif` last.
3. **Given** an existing locale-aware Noto Sans resolution path (e.g. `packages/fonts/src/resolver/getFontByLocale.ts`), **When** it selects a script-specific Noto Sans variant for the active locale, **Then** that variant remains the primary Noto Sans in the fallback chain so the system continues to honor the per-locale choice.
4. **Given** a CSforAll page (separate tenant), **When** it renders, **Then** its font stack is **not modified** by this feature.

### User Story 6 — Responsive Sizing Steps Through the Scale With a 16px Floor (Priority: P3)

A reader views a code.org marketing page at desktop, tablet, and mobile widths. Each viewport renders a typographic hierarchy where every heading is visibly larger than the next-smaller heading and body text. At no viewport does any heading shrink below the body base of 1rem (16px). Heading sizes "step down" through the 8-step scale as the viewport narrows — a large-display heading might be Display 4xl on desktop and Display 2xl on mobile — while smaller headings closer to body size do not step (because they would collide with the floor).

**Why this priority**: Desktop-first sizing without a floor produces mobile pages where small headings (H5/H6) shrink below body text or collapse into each other, breaking hierarchy. Tying the responsive ladder to the same 8-step scale (`xs`–`4xl`) keeps every viewport on the same vocabulary as authors and avoids unprincipled `clamp()` curves.

**Independent Test**: In Storybook, render every heading level (H1–H6) and the body default at five viewport widths (mobile-sm, mobile, tablet, desktop, desktop-lg). At each viewport, confirm: (a) no rendered font-size is below 1rem; (b) H1 > H2 > … > H6 ≥ body; (c) any heading that steps down at a smaller viewport never crosses below body.

**Acceptance Scenarios**:

1. **Given** an H1 at desktop, **When** the viewport narrows to mobile, **Then** the H1 renders at a smaller step in the Display scale (e.g. Display 2xl at mobile, Display 4xl at desktop) without collapsing below H2.
2. **Given** an H6 at desktop, **When** the viewport narrows to mobile, **Then** the H6 either keeps its desktop size or steps to a size that is still ≥ 1rem; it never renders below 1rem.
3. **Given** any viewport ∈ {mobile-sm, mobile, tablet, desktop, desktop-lg}, **When** all six heading levels render side-by-side, **Then** font-size is strictly non-increasing from H1 to H6 and each size is ≥ the body base.
4. **Given** the responsive step for a heading level is changed (e.g. H2 steps down on tablet rather than desktop), **When** the role token's breakpoint table is edited, **Then** every default H2 across the site reflects the new step without per-entry edits.

### Edge Cases

- **Existing Heading entries with deprecated `visualAppearance` values**: `heading-xxl/xl/lg/md/sm/xs` entries continue to render via an auto-map at the component layer — they map to the closest cell in the new (track, size, weight) grid. Authors do not re-publish.
- **Existing Paragraph entries with `body-one/two/three/four`**: Continue to render via an auto-map to the new Text-track size scale. No re-publish required.
- **Rich Text inline elements**: Inline `strong`, `em`, links, headings (`h1`–`h6` emitted by Contentful Rich Text), and paragraphs MUST resolve through the canonical role tokens, not parallel hardcoded values.
- **Hardcoded font-size in deprecated components**: `YourSchool` (2rem / 1.25rem / 0.875rem), `AdoptionMap` (0.75rem), and `AfeEligibility` (1rem / 0.625rem) are migrated to the shared role tokens or shared CSS variables — sub-1rem values are routed to the smallest Text-track size ≥ 1rem (the floor rule applies).
- **CJK / RTL / Devanagari glyph metrics**: The Noto Sans fallback chain produces different glyph heights than Space Grotesk / Geist. Line-heights are set in `rem` so a script with a tall x-height still fits within the role token's line-box; spot checks are part of the per-locale Storybook validation.
- **Letter-spacing**: Display-track sizes at the larger end (Display 2xl/3xl/4xl) often want tightened tracking. Each role token may carry an optional `letterSpacing` field; the default is `0` (or `normal`) when unspecified. Confirmed against Figma in `/plan`.
- **FOIT / FOUT**: Space Grotesk and Geist are loaded via the existing `packages/fonts` mechanism with `font-display: swap` semantics. The Noto Sans variants are already bundled by `packages/fonts` (Latin in the critical-path CSS, script-specific variants lazy-loaded per locale via `getFontByLocale` + `FontLoader`); this feature only adds those variant names into the MUI `font-family` cascade so the browser actually selects them when the primary lacks a glyph. **No new `@font-face` declarations and no new font downloads** are introduced.
- **Tenant scope**: CSforAll's typography theme is untouched. Both tenants load the shared `packages/component-library-styles/typography.module.scss` and `packages/fonts/src/`, so any change to those shared files MUST preserve CSforAll's existing rendered output exactly (a parallel set of code.org-only role tokens / variants is preferred over altering shared variables when there is any risk of CSforAll drift).
- **Storybook**: Stories MUST cover (a) each heading level's canonical default, (b) at least one per-instance override per Heading/Paragraph, (c) Rich Text headings + paragraphs, (d) at least one deprecated component (Hero Banner) showing the same canonical defaults, (e) the five viewport widths for the responsive ladder.
- **No new Contentful schema work**: All changes are application-code only (theme variants, component default resolution, SCSS variables, font-stack composition).
- **A11y**: The base body size remains 1rem (browser-default 16px), satisfying the legibility floor; responsive sizing never drops below this; Rich Text inherits the same chain so screen-reader / reflow behavior is unchanged.

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: The code.org typography theme (`apps/marketing/src/themes/code.org/index.ts`) MUST define MUI `theme.typography` variants for `h1`, `h2`, `h3`, `h4`, `h5`, `h6`, and `body1` (the body default), each bound to one canonical cell in the (track × size × weight) grid. `body2`, `body3`, `body4`, `overline`, `caption` MUST also bind to grid cells so the existing MUI variant surface stays usable without parallel definitions.
- **FR-002**: H1 MUST default to **Space Grotesk, Display 2xl, Semibold** (the only heading-level value the user has explicitly locked at spec time). The remaining heading defaults (H2–H6) MUST descend the Display scale in single-step increments, with weights and exact size cells confirmed against the Figma file in `/plan`. The default ladder MUST be specified in one file, not duplicated across components.
- **FR-003**: The body default (`theme.typography.body1` and equivalent) MUST resolve to **Geist, Text Md, Medium (weight 500), line-height 1.5rem (24px)** — taken from the Figma `text-md` token (font-size `16`, line-height `24`). The body base MUST remain `1rem` (16px) per the existing root font-size contract. (Note: an earlier draft stated 1.25rem / 20px for the line-height; the user has explicitly overridden that and pinned the value to Figma's Text Md.)
- **FR-004**: Two typographic tracks MUST be defined: a **Display** track using Space Grotesk and a **Text** track using Geist. Each track MUST expose **4 weights** (Regular / Medium / Semibold / Bold — final list confirmed in `/plan` against Figma) and **8 sizes** (`xs`, `sm`, `md`, `lg`, `xl`, `2xl`, `3xl`, `4xl`). Both tracks share the same size-token vocabulary so authors and developers can talk about "size 2xl" without specifying track.
- **FR-005**: All font sizes and line-heights MUST be expressed in `rem`. The body base MUST remain `1rem` (16px). No `px` literals for `font-size` or `line-height` in role tokens, the typography theme, or shared SCSS typography primitives.
- **FR-006**: Exact px / rem values for each (track, size) cell — including line-height and any letter-spacing — MUST be derived from the Figma file (`Aw6YXqpx6QFlNMXqCKk60e`, Space Grotesk node `36:874`, Geist node `36:975`), reading the **visible size label embedded in each sample text block** (`4xl`, `3xl`, `2xl`, `xl`, `lg`, `md`, `sm`, `xs` top-to-bottom) rather than the Figma variable names, which are mislabeled. The mapping is captured during `/plan` and recorded in research / data-model artifacts.
- **FR-007**: A **font-family fallback chain** MUST be applied to both tracks: `<primary track>, <Noto Sans family list, locale-aware via the existing packages/fonts/src/resolver/getFontByLocale.ts resolver>, sans-serif`. The Noto Sans list MUST cover the scripts the prior chain covered (Latin, Arabic, CJK JP / SC / TC, Korean, Devanagari, and any others restored from prior repo state); `sans-serif` MUST be the final fallback.
- **FR-008**: The Noto Sans chain MUST NOT introduce additional webfont downloads. The `packages/fonts` workspace **already bundles** Latin `Noto Sans` (via `@fontsource-variable/noto-sans`) into both tenants' critical-path font CSS and lazy-loads per-locale script-specific variants (Arabic / CJK / Devanagari / Hebrew / etc.) via the existing `getFontByLocale` resolver and the `FontLoader` client component. This feature **does NOT add new `@font-face` declarations or new font downloads** — it adds the same Noto Sans variant names that the `packages/fonts` workspace already declares to the MUI `font-family` cascade so the browser actually resolves to them when the primary face lacks a glyph. The full chain target replicates the 21-variant version that lived in `themes/code.org/index.ts` at commit `984a9f05` (2025-06-27) before being stripped in subsequent CSforAll refactor and Geist swap commits — with `Geist` swapped in as the primary for body and `Space Grotesk` swapped in as the primary for headings.
- **FR-009**: The `Heading` component (`apps/marketing/src/components/contentful/heading/Heading.tsx`) MUST render with a deterministic three-step style-resolution chain:
  1. **Heading Level default** (from the existing `visualAppearance` field, Studio displayName "Heading Level", enum `heading-xxl`/`xl`/`lg`/`md`/`sm`/`xs`) — selects the semantic tag (`h1`…`h6` per the 1:1 legacy map) AND seeds the canonical role token (family, size, weight, line-height, letter-spacing, per-breakpoint steps) from `ROLE_TOKENS[h{N}]`.
  2. **Visual Appearance override** (new field `appearance`, Studio displayName "Visual Appearance", enum `default` + 8 Display cells `display-xs`/`-sm`/`-md`/`-lg`/`-xl`/`-2xl`/`-3xl`/`-4xl`, defaultValue `default`) — when non-`default`, replaces the size + weight + line-height + letter-spacing + per-breakpoint steps from step 1 with the chosen Display cell's canonical role token (Semibold weight by default). The **semantic tag from step 1 is preserved unchanged.**

### Key Entities

- **Typographic Track**: One of two pre-defined family + scale namespaces — **Display** (Space Grotesk) and **Text** (Geist). Each track has a fixed font-family fallback chain and 8 size cells.
- **Size Token**: One of `xs`, `sm`, `md`, `lg`, `xl`, `2xl`, `3xl`, `4xl`. Shared vocabulary across both tracks; the same token name resolves to different rem values per track.
- **Weight Token**: One of 4 named weights (Regular / Medium / Semibold / Bold — final list confirmed in `/plan`).
- **Role Token**: A named binding from a semantic role (`h1`, `h2`, `h3`, `h4`, `h5`, `h6`, `body1`, `body2`, etc.) to a (track, size, weight) tuple, plus a line-height, an optional letter-spacing, and an optional per-breakpoint step table. The canonical surface for site-wide defaults.
- **Fallback Chain**: An ordered font-family list — `<primary track>, <Noto Sans family list (locale-aware)>, sans-serif` — applied uniformly to both tracks. Identical structure for Display and Text; only the primary differs.
- **Legacy `visualAppearance` value**: Existing Heading values `heading-xxl/xl/lg/md/sm/xs` and Paragraph values `body-one/two/three/four`. Mapped at the component layer to role tokens at render time so existing Contentful entries continue to render unchanged.
- **Heading Visual Appearance value**: New Heading-specific enum on the `appearance` ComponentDefinition variable — the sentinel `default` plus 8 Display cells (`display-xs` / `display-sm` / `display-md` / `display-lg` / `display-xl` / `display-2xl` / `display-3xl` / `display-4xl`). `default` means "use the Heading Level's canonical role token unchanged." Each Display cell carries a canonical role token (Display track + cell size + Semibold weight + cell line-height + cell letter-spacing + the cell's own per-breakpoint step table).
- **Paragraph Visual Appearance value (new)**: Eight new Text cells (`text-xs` / `text-sm` / `text-md` / `text-lg` / `text-xl` / `text-2xl` / `text-3xl` / `text-4xl`) added to the existing enum. Each maps to a canonical role token (Text track + cell size + canonical weight + cell line-height). Legacy `body-*` values remain valid and keep their existing bindings.
- **Style Resolution Chain**: The deterministic three-step (Heading) / two-step (Paragraph) procedure that resolves what styles end up on the rendered element. Heading: Heading Level default → Visual Appearance override → individual field overrides. Paragraph: Visual Appearance default → individual field overrides. Each step touches only the dimensions the next step does not override.
- **Deprecated component**: A code.org-only component that currently hardcodes its typography (Hero Banner, `YourSchool`, `AdoptionMap`, `AfeEligibility`, etc.). Migrated to consume the role tokens via theme variants or shared SCSS variables; no new tokens are introduced specifically for them.
